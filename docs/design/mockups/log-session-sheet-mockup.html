Log a session — restyled sheet, keyboard-aware

A re-dress of the shipped progress-logging sheet in the "Add a past session" aesthetic. The scope is exactly what ships today: the From → To range (From = where you left off, the reference; To = the value you type), a quiet date defaulting to today, and four actions — Save · I finished the book · Give up · Cancel. No unit toggle, no notes, no multi-format.

The keyboard fork is panel 3 (keep the OS keyboard and lift the sheet so From→To + date + Save stay parked above it) vs panel 4 (a custom number pad inside the sheet, no OS keyboard). Chosen: panel 3 — the OS keyboard; panel 4 is kept as a parked fast-follow in case the native keyboard doesn't feel right. Everything else is shared.

Notes on the two open threads

Keyboard (panels 2 vs 3). Panel 2 keeps the native keyboard and lifts the sheet using the visualViewport API so the From→To box and Save stay docked above it; the three secondary actions step aside while typing. Panel 3 removes the OS keyboard entirely with an in-sheet pad, so nothing is ever occluded and all actions can stay — at the cost of a bigger build and a different entry feel. This is the decision to make from these pixels.

Device chrome is not fixed. These panels model the gesture-pill nav bar (transparent strip at the very bottom) and a representative Chrome address bar, but both vary by device: 3-button nav eats a solid ~48px bar instead of the pill, and some Chrome builds add a home button / taller bar. So the sheet must tolerate a range of top-and-bottom chrome, not a single measured height — the lifted layout leaves margin for that, and nothing load-bearing sits in the outermost few pixels.

Back button — implementation note (verify during build). Goal: the Android system back button closes the sheet and returns to Currently Reading, without giving the sheet its own route. Approach to try: on open, push a throwaway history entry (history.pushState); listen for popstate and, when back fires it, dismiss the sheet instead of navigating; on a normal close (Save / Cancel / ✕), pop that entry back off so history stays clean. Unverified against Angular's router + Material bottom-sheet/overlay — test this early, before relying on it.

Decided: To opens empty and unfocused. Today's shipped sheet pre-fills the single input with your last position. Because From now shows that reference on its own, To opens empty (panels 1 & 4). It also opens unfocused, so the soft keyboard does not auto-pop on open — the sheet appears calm and the keyboard rises only when you tap To. Together this kills two paper cuts (the "clear the box first" edit and the keyboard covering the sheet on open). Implementation: Material autofocuses the first focusable control by default (which would summon the keyboard) — disable it (e.g. autoFocus: false on the bottom-sheet / dialog config) so the keyboard stays down until To is tapped.
